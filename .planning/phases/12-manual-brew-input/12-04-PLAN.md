1. Add `POST /brew/extend-ranges` route that accepts the active bean and new bound values:
   ```python
   @router.post("/extend-ranges", response_class=HTMLResponse)
   async def extend_ranges(
       request: Request,
       db: Session = Depends(get_db),
   ):
       """Extend bean parameter ranges when manual input exceeds current bounds."""
       bean = _require_active_bean(request, db)
       if not bean:
           return RedirectResponse(url="/beans", status_code=303)

       form = await request.form()
       overrides = dict(bean.parameter_overrides or {})

       from app.services.optimizer import DEFAULT_BOUNDS

       for param in DEFAULT_BOUNDS:
           new_min = form.get(f"{param}_min")
           new_max = form.get(f"{param}_max")
           if new_min is not None or new_max is not None:
               current = overrides.get(param, {})
               default_lo, default_hi = DEFAULT_BOUNDS[param]
               if new_min is not None:
                   current["min"] = float(new_min)
               if new_max is not None:
                   current["max"] = float(new_max)
               overrides[param] = current

       bean.parameter_overrides = overrides
       db.commit()

       # Return 200 OK — JS will proceed with form submission
       from fastapi.responses import JSONResponse
       return JSONResponse(content={"status": "ok"})
   ```

**Manual form modifications (manual.html):**

1. The number inputs should NOT have `min` and `max` HTML attributes (so users CAN type values outside the range). Only the range SLIDERS have min/max (they constrain the slider track).

2. Add `data-min` and `data-max` attributes to each number input so JS can check bounds:
   ```html
   <input type="number" id="num-grind_setting" name="grind_setting"
     class="param-number form-input"
     step="0.5" value="{{ defaults.grind_setting }}"
     data-min="{{ bounds.grind_setting.0 }}" data-max="{{ bounds.grind_setting.1 }}"
     data-param="grind_setting"
     oninput="document.getElementById('slider-grind_setting').value = this.value">
   ```
   (The `data-param` and `data-min`/`data-max` are used by the range extension JS.)

3. When the number input goes outside the slider range, the slider should clamp to its min/max (that's natural behavior), but the number input keeps the typed value. The form submits the number input's value (since `name` is on the number input).

4. Update the bidirectional sync: slider → number (always), number → slider (clamp to slider range):
   ```javascript
   // Slider updates number
   oninput="document.getElementById('num-{param}').value = this.value"
   // Number updates slider (clamped)
   oninput="var s = document.getElementById('slider-{param}'); s.value = Math.max(s.min, Math.min(s.max, this.value))"
   ```

**Out-of-range detection JS (in manual.html scripts block or tags.js):**

5. Add a form submit interceptor (in manual.html's script block, before the tags.js import) that:
   a. Collects all `.param-number` inputs
   b. For each, compares the current value against `data-min` and `data-max`
   c. If any are out of range, builds a message listing all violations:
      - "Extend grind range to {new_min}–{new_max} and temperature to {new_min}–{new_max}?"
      - new_min = min(current_min, typed_value), new_max = max(current_max, typed_value)
   d. Shows `confirm()` dialog with the message
   e. If user confirms: calls `fetch('/brew/extend-ranges', { method: 'POST', body: formData })` with the new min/max values, then allows form submission
   f. If user cancels: prevents submission (return false)
   g. If all values are in range: allows normal submission

6. Important: this check must run BEFORE the taste slider validation in tags.js's `initFlavorSliders` submit handler. The simplest approach is to add the check as a separate event listener on the form element in manual.html (with `{capture: true}` or call order that runs first), OR add it to tags.js as a named function that manual.html can configure.

   Recommended approach: Add the out-of-range check inline in manual.html's script block as an event listener on the specific form. It runs before tags.js's generic submit handler because it's registered first:
   ```javascript
   document.addEventListener('DOMContentLoaded', function() {
     var form = document.querySelector('form[action="/brew/record"]');
     if (!form || !document.getElementById('is_manual_flag')) return;

     form.addEventListener('submit', function(e) {
       var outOfRange = [];
       var extendData = new FormData();
       form.querySelectorAll('.param-number').forEach(function(input) {
         var val = parseFloat(input.value);
         var min = parseFloat(input.dataset.min);
         var max = parseFloat(input.dataset.max);
         var param = input.dataset.param;
         if (val < min || val > max) {
           var newMin = Math.min(min, val);
           var newMax = Math.max(max, val);
           outOfRange.push(param.replace(/_/g, ' ') + ' to ' + newMin + '–' + newMax);
           extendData.append(param + '_min', newMin);
           extendData.append(param + '_max', newMax);
         }
       });

       if (outOfRange.length > 0) {
         e.preventDefault();
         if (confirm('Extend ' + outOfRange.join(' and ') + '?')) {
           fetch('/brew/extend-ranges', { method: 'POST', body: extendData })
             .then(function() { form.submit(); });
         }
       }
     });
   });
   ```

7. Add a hidden input `<input type="hidden" id="is_manual_flag" value="true">` to manual.html's form (this is separate from the `name="is_manual"` hidden input — it's just a marker so the JS knows this is the manual form and should check ranges).
